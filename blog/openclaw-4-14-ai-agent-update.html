GPT 5.4 Reliability Fixes

If you've been running OpenClaw with GPT 5.4, you've likely experienced the most frustrating failure mode - agents freezing mid-task with no error message, no recovery, just dead silence. This happened when GPT 5.4 returned empty "reasoning-only" responses during complex tasks.

OpenClaw 4.14 completely rewrites how these empty responses are handled. Instead of stalling, agents now detect empty turns, automatically retry them with adjusted parameters, and maintain workflow continuity. The system also includes forward compatibility for GPT 5.4 Pro, meaning when that model launches, your existing OpenClaw setup will work immediately without requiring another update.

Impact: For teams using GPT 5.4 with OpenClaw, this single fix reduces agent failure rates by an estimated 68% based on community testing. No more manually restarting frozen workflows at 2 AM.

Subagent Reliability

Multi-agent architectures suffered from a particularly insidious bug - subagents would appear to queue properly but never actually launch. The root cause was a missing file (subagent_registry_runtime.js) that wasn't being included in npm builds.

Without this critical registry file, the system would fail silently when trying to spawn subagents. The parent agent would continue running, unaware its child agents were stuck in permanent limbo. Version 4.14 ensures this file is properly packaged, finally making complex agent hierarchies reliable.

Before/After: Pre-4.14, only 23% of subagent launches succeeded in production environments. Post-update, success rates exceed 98% with proper error handling for the remaining 2%.

LLaMA Timeout & Streaming Fixes

Local LLaMA deployments had two critical issues addressed in this release. First, custom timeout settings were being ignored entirely - your carefully configured 60-second timeout for slower local models was being overridden by a 30-second default.

Second, LLaMA streaming completions were reporting incorrect token counts, triggering premature memory compaction. This caused agents to lose context mid-conversation as the system falsely believed the context window was full. Both issues are now resolved, with proper timeout enforcement and accurate streaming token reporting.

Browser Integration Improvements

The security feature designed to prevent server-side request forgery (SSRF) had become its own worst enemy - blocking OpenClaw from communicating with its own Chrome browser instances. Agents would launch browsers successfully, then immediately lose connection due to overzealous security policies.

Version 4.14 rebalances these protections, maintaining strict security for external sites while allowing normal browser operation. Navigation works reliably again, and strict mode remains available for high-security environments.

Messaging Platform Fixes

Telegram integration received two critical fixes. First, agents can now properly identify forum topic names (not just IDs), giving them crucial context about conversation location. Second, this topic knowledge persists across restarts rather than resetting.

Slack now properly enforces allow lists across all interaction types, closing a security gap where blocked users could still trigger actions through buttons and modals. Discord shows correct status responses instead of generic DOM messages when using native commands.

Security Improvements

Multiple AI-assisted security audits identified and fixed vulnerabilities, including:

  • Preventing dangerous config flags from being enabled through patches
  • Proper SSRF enforcement for browser screenshots
  • Allow list verification for Microsoft Teams SSO events
  • Complete blocking of unresolved media attachment paths

The control UI also replaced its markdown library (mark.js) with the more secure markdown-it after identifying potential vulnerabilities in how user-generated content was rendered.

Memory System Fixes

Three critical memory issues were resolved:

  1. LLaMA embeddings: Now work correctly when explicitly set as provider, with proper cache key separation between instances
  2. QMD memory: No longer searches phantom lowercase "memory_md" collections that didn't exist
  3. Dreaming system: Fixed memory reprocessing during subsequent heartbeats after completion

Together, these fixes make agent memory more reliable and contextually accurate across long-running conversations.

Voice Call Reliability

Voice agents received two important fixes for production deployments:

  • Resolved race condition where concurrent WebSocket handshakes could exceed connection limits
  • Proper source IP detection when behind proxies, improving security logging

These changes are particularly valuable for contact center implementations where multiple voice agents may be handling concurrent calls.

What This Workflow Does

In software development and IT support, timely awareness of new bugs is critical. Without automation, teams must manually check MantisBT or rely on email notifications, which are often missed or delayed. This creates bottlenecks in issue resolution, slows down development cycles, and can lead to customer dissatisfaction when critical problems aren't addressed quickly.

This Make.com workflow solves that by creating a real-time bridge between your bug tracking system and your team's communication hub. It continuously monitors your MantisBT instance for newly created issues. As soon as one is detected, it automatically formats a detailed alert and posts it directly to a designated Slack channel. This ensures every relevant team member—developers, QA engineers, project managers—is notified instantly, within the tool they use daily.

The automation transforms a passive tracking system into an active alerting system. It not only saves the time spent on manual monitoring but also improves accountability and collaboration by bringing issue data into the conversational space of Slack, where discussions, assignments, and prioritizations happen naturally.

How It Works

The workflow operates on a simple, reliable polling principle, orchestrated by Make.com's visual automation platform.

Step 1: Scheduled Monitoring

A scheduled module in Make.com acts as the trigger. It is configured to check your MantisBT API at regular intervals (e.g., every 5 or 10 minutes). This module searches for issues created since the last check, filtering based on your project, category, or status if needed.

Step 2: Data Retrieval & Processing

When a new issue is found, Make.com retrieves its full details: issue ID, title, description, priority, status, reporter, and any custom fields. This data is parsed and prepared for the next step. You can add filters here to ignore low-priority issues or only alert for specific projects.

Pro tip: Use Make.com's filter tool to only send Slack alerts for high or critical priority issues, preventing notification overload for minor bugs.

Step 3: Slack Message Creation & Delivery

The processed issue data is passed to the Slack module. Make.com constructs a clear, actionable message. This typically includes a direct link to the MantisBT issue, key details, and can even mention specific team members based on the assignee or project. The message is then posted to your chosen Slack channel, ensuring immediate visibility.

The entire process runs silently in the background, requiring zero ongoing manual intervention. It creates a closed-loop system where a bug report in MantisBT triggers a team alert in Slack within minutes.

Quick Setup Guide

After importing the template into your Make.com account, follow these steps to activate it:

  1. Connect MantisBT: In the first module, enter your MantisBT base URL and authentication details. Test the connection to ensure Make.com can fetch issues.
  2. Set the Schedule: Configure the schedule interval. For active projects, polling every 5-10 minutes is typical. Adjust based on your issue volume.
  3. Connect Slack: In the Slack module, authorize Make.com with your Slack workspace and select the target channel (e.g., #bugs, #dev-alerts).
  4. Customize the Message: Edit the message template in the Slack module. Include the most useful info: Issue ID, title, priority, and a direct link back to MantisBT.
  5. Add Filters (Optional): Insert a filter between modules to only process issues matching certain criteria (priority = "high", project = "Website").
  6. Turn It On: Save the scenario and activate it. The workflow will now run on its schedule, monitoring and alerting automatically.

What This Workflow Does

Managing GitHub issues in open-source projects or development teams often creates a bottleneck. Maintainers spend hours manually triaging issues, assigning them to contributors, and responding to volunteer requests. This manual process slows down development, frustrates eager contributors, and prevents projects from scaling effectively.

This automation solves that problem by creating a self-service assignment system. When contributors comment "assign me" on an issue, the workflow automatically assigns them to it, adds appropriate labels, and prevents conflicts. It monitors both new issues and comments, creating a fair, first-come-first-served system that eliminates manual intervention while maintaining proper tracking.

The result is a streamlined project where maintainers focus on code review and project direction instead of administrative tasks, contributors get immediate feedback when they volunteer, and issues move forward faster without waiting for human assignment.

How It Works

1. Trigger Monitoring

The workflow uses GitHub webhooks to monitor your repository for two events: new issues being created and new comments being added to existing issues. It filters these events to detect when someone includes the phrase "assign me" in either the issue description or a comment.

2. Assignment Validation

Before making any assignment, the workflow checks if the issue is already assigned to someone else. It fetches the current issue details from GitHub and verifies the assignee field is empty. This prevents conflicts and duplicate assignments.

3. Automatic Assignment

If the issue is unassigned, the workflow automatically assigns it to the user who commented "assign me" or created the issue with that phrase. It uses GitHub's API to update the issue assignee field with the appropriate user.

4. Label Management

After successful assignment, the workflow adds an "assigned" label to the issue for better tracking and visibility. This helps maintainers and other contributors quickly identify which issues are already being worked on.

5. Conflict Response

If someone tries to claim an already-assigned issue, the workflow posts a polite comment explaining that the issue is already taken and suggesting they look for other unassigned issues. This maintains positive contributor relationships.

Quick Setup Guide

Follow these steps to implement this automation in your project:

  1. Download the template using the button above and import it into your n8n instance.
  2. Configure GitHub credentials in n8n with appropriate repository access permissions.
  3. Update the repository name in the GitHub trigger node to point to your target repo.
  4. Set up the webhook in your GitHub repository settings if not automatically configured.
  5. Test the workflow by creating an issue with "assign me" or commenting on an existing issue.
  6. Monitor the first few assignments to ensure everything works as expected.
  7. Communicate the new process to your contributors so they know how to claim issues.

What This Workflow Does

Manual sprint reporting is a time-consuming, error-prone task that pulls Scrum Masters and Product Owners away from actual product work. This automation solves that by connecting Jira directly to your email system.

The workflow runs on a schedule (like every Friday at 5 PM), fetches all active sprint issues from Jira, validates and cleans the data, calculates key performance indicators, generates a polished HTML report, and emails it to stakeholders. It turns hours of manual compilation into a fully automated, consistent process.

Beyond just saving time, it ensures data accuracy, provides stakeholders with professional updates on time every time, and creates a searchable archive of sprint performance for retrospectives and planning.

How It Works

1. Scheduled Trigger

The workflow activates on a predefined schedule—typically at the end of each week or sprint. You set it once and it runs automatically forever.

2. Jira Data Fetch

Using Jira's API with a secure token, the workflow executes a JQL query to pull all issues from the current active sprint. It retrieves key fields: issue key, summary, status, assignee, priority, story points, and sprint details.

3. Data Validation & Normalization

Raw Jira data is cleaned: empty story points are set to zero, statuses are standardized, assignee names are formatted, and invalid records are filtered out to ensure report accuracy.

4. Metrics Calculation

The workflow aggregates the data to calculate: total tickets, completed tickets, completion percentage, total story points, points delivered, number of blockers, and progress against sprint goals.

5. HTML Report Generation

A professional, mobile-friendly HTML template is populated with the metrics and a detailed issue table. The report includes direct links back to Jira issues for easy navigation.

6. Email Delivery via Gmail

The completed HTML report is sent as an email to your configured stakeholder list via Gmail's API. The email subject includes the sprint name and date for easy identification.

What You'll Need

  1. Jira Cloud instance with API access and a project containing sprints.
  2. Jira API token with read permissions for issues and sprints.
  3. Gmail account or Google Workspace account with API access enabled.
  4. n8n instance (cloud or self-hosted) to run the workflow.
  5. Stakeholder email list of who should receive the reports.
  6. Basic understanding of your team's Jira project structure and sprint naming conventions.

Pro tip: Create a dedicated "Sprint Reports" label in Gmail and configure the workflow to apply it automatically. This keeps all historical reports organized and searchable in one place.

Quick Setup Guide

  1. Download the template using the button above and import it into your n8n instance.
  2. Configure the Schedule Trigger to match your reporting cadence (e.g., every Friday at 17:00).
  3. Set up Jira credentials with your base URL and API token in the "Get Many Issues" node.
  4. Update the JQL query to match your project key and any custom filters your team uses.
  5. Configure Gmail credentials and enter your stakeholder email addresses in the "Send Email" node.
  6. Test with a dry run by manually executing the workflow and checking the output.
  7. Activate the workflow and let it run automatically according to your schedule.

Backup n8n Workflows with Versioning & Notion Tracking

Automatically copy workflows between n8n instances with date-based versioning and centralized audit logs in Notion.

What This Workflow Does

This automation solves a critical problem for businesses relying on n8n for their operations: the risk of losing valuable workflow logic due to server failures, accidental deletions, or migration errors. Manual backups are often forgotten, leaving companies vulnerable to hours of recreation work and potential revenue loss.

The workflow automatically copies all workflows from a source n8n instance to a destination (backup) instance, prefixing each with the current date (YYYY_MM_DD_) to maintain multiple versions. It then logs the execution date and number of workflows processed into a Notion database, creating a centralized audit trail. This ensures you always have recoverable versions of your automation logic while maintaining clear visibility into your backup operations.

How It Works

The automation follows a systematic process to ensure reliable backup operations with proper version management.

1. Source Workflow Retrieval

The workflow begins by connecting to your primary n8n instance using its API credentials. It retrieves all active workflows, including their complete configuration, nodes, connections, and settings. This creates a comprehensive snapshot of your current automation estate.

2. Date-Based Versioning

Each retrieved workflow is renamed with the current date prefix (e.g., "2025_03_15_Invoice Processor"). This simple yet powerful approach allows you to maintain multiple historical versions on the destination instance, enabling rollback to any previous day's version if needed.

3. Destination Workflow Creation

The renamed workflows are then created on your backup n8n instance. The automation handles all API communications and error scenarios, ensuring each workflow is properly replicated with its complete configuration intact.

4. Notion Audit Logging

After processing all workflows, the automation creates a detailed entry in your Notion database. This includes the execution timestamp, number of workflows successfully backed up, and any relevant notes about the operation. This creates a searchable history of all backup activities.

5. Retention Policy Enforcement

The workflow can be configured to automatically delete older versions beyond your retention period (default 2 days). This prevents storage bloat while maintaining sufficient historical coverage for recovery needs.

What You'll Need

  1. Two n8n instances – A source (production) and destination (backup) n8n installation, either self-hosted or cloud.
  2. API credentials for both n8n instances with permissions to read (source) and create/delete (destination) workflows.
  3. A Notion account with a database created containing fields for date, workflow count, and execution notes.
  4. Basic n8n knowledge to configure credentials and understand workflow import/export concepts.
  5. Network connectivity between your n8n instances and Notion's API endpoints.

Pro tip: Before running this in production, test with a small subset of workflows and verify you can successfully restore from your backup instance. Also, secure your API keys using n8n's credential management—never hardcode them in the workflow.

Quick Setup Guide

  1. Download and import the template into your n8n instance using the Import from File function.
  2. Create your Notion database with three fields: "sequence" (containing "prefix"), "Value" (for the date), and "Comment" (for workflow count).
  3. Configure the Notion node with your integration token and database ID from Notion's API settings.
  4. Set up n8n API credentials for both source and destination instances in n8n's credential management.
  5. Adjust retention settings in the "Subtract From Date" node based on how many days of versions you want to keep.
  6. Test with limit by setting the Limit node to 1-2 workflows initially to verify everything works.
  7. Schedule the workflow to run daily or weekly based on your change frequency and recovery requirements.

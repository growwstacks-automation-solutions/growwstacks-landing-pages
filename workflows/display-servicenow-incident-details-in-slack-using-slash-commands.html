ServiceNow-Slack Incident Lookup Automation

Fetch ServiceNow ticket details directly in Slack with slash commands

Download Template JSON · n8n compatible · Free

What This Workflow Does

This automation bridges ServiceNow incident management with Slack communication, allowing team members to instantly retrieve ticket details without switching between applications. By using a simple slash command in Slack, users can query ServiceNow for incident information, reducing resolution times and improving collaboration.

The workflow handles authentication, data transformation, and error handling to ensure secure and reliable incident lookups. It automatically formats the response in Slack with key details like priority, status, description, and assignee information.

How It Works

1. Slack Command Trigger

The workflow initiates when a user types a predefined slash command in Slack (e.g., /incident). The command includes the ServiceNow incident number as a parameter.

2. ServiceNow API Query

Using the provided incident ID, the workflow queries ServiceNow's REST API to retrieve the full ticket details. It handles authentication and rate limiting automatically.

3. Data Processing

The raw ServiceNow response is parsed to extract relevant fields and transformed into a user-friendly format optimized for Slack's message interface.

4. Slack Response

The formatted incident details are posted back to the originating Slack channel as an ephemeral message visible only to the requester, maintaining data security.

  1. A ServiceNow instance with API access

  3. n8n instance (cloud or self-hosted)
  4. ServiceNow API credentials with read access to incidents

Quick Setup Guide

  1. Download and import the JSON template into your n8n instance
  2. Configure ServiceNow API credentials in the designated node
  3. Set up the Slack webhook trigger with your slash command
  4. Test with sample incident IDs to verify formatting
  5. Deploy the workflow and train your team on the new command

Pro tip: Create multiple slash commands for different ticket types (e.g., /incident, /change, /problem) by duplicating and modifying this workflow.

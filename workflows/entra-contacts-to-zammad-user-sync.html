What This Workflow Does

Manual user management between your identity provider and helpdesk system creates security gaps, delays in support access, and administrative overhead. This automation solves that by creating a seamless, scheduled sync between Microsoft Entra (formerly Azure AD) and Zammad.

The workflow automatically fetches user and contact data from Entra, transforms it into a Zammad-compatible format, and performs intelligent synchronization. It identifies new employees who need Zammad accounts, updates existing user profiles with current information, and can deactivate accounts for departed employees—all without manual intervention.

By automating this process, you ensure your support team always works with accurate, up-to-date user information while maintaining security compliance and reducing IT administration time by hours each week.

How It Works

The automation follows a logical three-stage process to ensure data integrity and proper user management.

1. Fetch Entra Contacts & Users

The workflow connects to Microsoft Graph API to retrieve all active users and contacts from your Entra directory. It handles pagination for large directories and filters based on your configured criteria (such as department, employee type, or group membership).

2. Create Universal User Object

Raw Entra data is transformed into a standardized format that Zammad understands. This step maps Entra attributes (displayName, mail, telephoneNumber, department) to corresponding Zammad user fields, including any custom field mappings you've configured.

3. Synchronize with Zammad

The workflow performs intelligent matching using email addresses as the primary key. For each Entra record, it checks if a corresponding Zammad user exists. If not, it creates a new user. If yes, it updates the existing profile. Optionally, it can identify users in Zammad who no longer exist in Entra and deactivate them.

Pro tip: Schedule this workflow to run daily during off-hours (e.g., 2 AM) to minimize impact on system performance and ensure morning readiness for your support team.

What You'll Need

  1. Microsoft Entra/Azure AD access with Application Administrator or similar permissions to create an app registration and grant API permissions
  2. Zammad instance with administrator access to create API tokens and custom user fields
  3. n8n instance (self-hosted or cloud) with network access to both Entra and Zammad APIs
  4. Custom field in Zammad named "entra_key" (type: String) to store the Entra object identifier
  5. Optional custom field "entra_object_type" (Single selection: User/Contact) if syncing both user types
Configuration of selection field in Zammad to distinguish between User and Contact types from Entra

Quick Setup Guide

  1. Download the template using the button above and import it into your n8n instance
  2. Configure Microsoft Entra credentials in n8n using OAuth2 with User.Read.All and Directory.Read.All permissions
  3. Set up Zammad API token with user read/write permissions in your Zammad instance
  4. Create the custom fields in Zammad User object as shown in the screenshots above
  5. Test with a small user group first by modifying the filter in the "Fetch Entra Contacts" node
  6. Schedule the workflow to run automatically at your preferred interval (daily recommended)
  7. Monitor the first few runs and adjust field mappings if needed for your specific requirements

Security note: Store your API credentials securely using n8n's credential system rather than hardcoding them in the workflow. Use the principle of least privilege when granting API permissions.

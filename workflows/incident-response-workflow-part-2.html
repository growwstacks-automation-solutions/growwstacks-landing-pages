Incident Response Workflow Part 2: Automated Acknowledgment

Automatically update incident status in PagerDuty and notify teams in Mattermost when an incident is acknowledged—reducing response time by 80%.

Download Template JSON · n8n compatible · Free

What This Workflow Does

This workflow is the second part of a three-part incident response automation series. It specifically handles the acknowledgment phase of incident management—the critical moment when an engineer first engages with an alert. When someone clicks the "Acknowledge" button in your monitoring system, this workflow automatically updates the incident status in PagerDuty from 'Triggered' to 'Acknowledged' and simultaneously posts a notification in your Mattermost team channel.

The automation eliminates manual status updates and team notifications, ensuring everyone knows an incident is being addressed within seconds. This reduces mean time to acknowledge (MTTA) by 80-90%, prevents duplicate work, and creates a clear audit trail of incident response activities.

How It Works

This three-node workflow creates a seamless bridge between your alerting system and team communication platform.

Step 1: Webhook Trigger

The workflow starts with a webhook node that listens for the acknowledgment event. When someone clicks "Acknowledge" in your monitoring interface (or when an automated system triggers acknowledgment), it sends a webhook payload containing incident details like ID, title, and who acknowledged it.

Step 2: PagerDuty Status Update

The workflow takes the incident ID from the webhook and uses the PagerDuty node to change the incident status from 'Triggered' to 'Acknowledged'. This ensures your incident management system reflects the current state in real-time, preventing confusion about whether someone is working on the issue.

Step 3: Mattermost Team Notification

Finally, the Mattermost node posts a formatted message to your designated team channel. The message includes the incident title, who acknowledged it, timestamp, and a direct link to the incident in PagerDuty. This keeps the entire team informed without requiring manual updates or @mentions.

What You'll Need

  1. n8n instance (cloud or self-hosted) with workflow execution capabilities
  2. PagerDuty account with API access and existing incidents configured
  3. Mattermost workspace with webhook permissions for the target channel
  4. Webhook source that can trigger when incidents are acknowledged (this could be from your monitoring tool, a custom application, or even Part 1 of this workflow series)
  5. Basic understanding of how to configure API credentials in n8n

Quick Setup Guide

  1. Download and import the JSON template into your n8n instance using the import function.
  2. Configure the Webhook node by copying the webhook URL provided by n8n and setting it up in your monitoring system to trigger when incidents are acknowledged.
  3. Set up PagerDuty credentials in n8n using an API key from your PagerDuty account with permissions to update incidents.
  4. Configure Mattermost credentials using a webhook URL from your Mattermost channel (create via Integrations → Incoming Webhooks).
  5. Test the workflow by acknowledging a test incident in your monitoring system and verifying that PagerDuty updates and Mattermost notifications work correctly.
  6. Activate the workflow and monitor initial executions to ensure all connections are working as expected.

Pro tip: Use this workflow in conjunction with Part 1 (incident creation) and Part 3 (incident resolution) to create a complete, automated incident response lifecycle. Connect all three workflows for end-to-end automation from alert to resolution.

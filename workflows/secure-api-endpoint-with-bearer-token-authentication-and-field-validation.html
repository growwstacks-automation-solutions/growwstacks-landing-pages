What This Workflow Does

Exposing public API endpoints or webhooks is powerful but risky. Without proper security, anyone can send data to your systems, leading to potential data breaches, spam, or system overload. This workflow solves that by acting as a secure gateway.

It intercepts incoming HTTP requests, verifies a secret Bearer Token in the Authorization header, and checks that all required data fields are present and valid. Only authenticated and well-formed requests proceed to your business logic, while unauthorized or malformed requests receive immediate, standardized error responses.

This template provides a reusable foundation for building secure, API-style workflows within n8n, eliminating the need to write custom authentication code for every integration.

How It Works

The workflow follows a logical security chain. Each step acts as a gatekeeper, ensuring only legitimate data flows through.

Step 1: Webhook Trigger & Initial Receipt

The workflow is triggered by an incoming HTTP POST request to a public n8n webhook URL. The raw request body and headers are captured for processing.

Step 2: Bearer Token Authentication Check

The workflow extracts the Authorization header and compares it against a pre-configured secret token stored securely within the workflow. If the header is missing or the token doesn't match, the workflow immediately stops and returns a 401 Unauthorized JSON response.

Step 3: Dynamic Field Validation

Once authenticated, the request body is parsed (usually JSON). The workflow checks for the presence of specific, required fields (e.g., email, message, userId). You define which fields are mandatory in a simple configuration node. If any are missing, the workflow returns a 400 Bad Request error detailing what's wrong.

Step 4: Processing & Success Response

If the request passes both authentication and validation, it proceeds to your custom business logic (which you add). Finally, a standardized 200 OK JSON success response is sent back to the client, confirming receipt and processing.

Pro tip: Store your secret Bearer Token in n8n's credentials system or environment variables instead of hardcoding it in the workflow for better security and easier rotation.

What You'll Need

  1. A running n8n instance (Cloud, Self-Hosted, or Desktop).
  2. A secret Bearer Token that you will share with authorized clients/systems.
  3. Knowledge of the expected data structure (the fields your endpoint requires).
  4. The ability to configure HTTP clients (like Postman, cURL, or another app) to send requests with an Authorization: Bearer YOUR_TOKEN header.

Quick Setup Guide

Get this security layer up and running in your n8n environment in just a few minutes.

  1. Download & Import: Click the download button above to get the JSON file. In your n8n editor, go to Workflows → Import from File and select the downloaded template.
  2. Configure the Security Settings: Locate the "Configuration" node in the workflow. Open it and set your bearerToken value. Then, define the requiredFields object by listing the keys your application expects (e.g., email: true, name: true).
  3. Activate the Webhook: Select the "Webhook" trigger node and click Execute Node to generate your unique, public webhook URL. Copy this URL.
  4. Add Your Business Logic: Between the validation and the success response, add your own n8n nodes to process the data (e.g., save to a database, send a notification, update a CRM).
  5. Test Thoroughly: Use a tool like Postman to send test requests with and without the correct token and required fields. Verify you receive the proper 401, 400, and 200 responses.

API keys are often passed as query parameters or headers and identify a project/application. Bearer Tokens are a specific type of token passed in the Authorization header and represent the authorization granted to a client. Both secure APIs, but Bearer Tokens follow a standard OAuth 2.0 pattern.

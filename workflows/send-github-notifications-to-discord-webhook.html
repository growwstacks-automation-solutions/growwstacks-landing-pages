What This Workflow Does

GitHub's native mobile notifications are limited—they only alert you for direct @mentions, leaving you in the dark about important code changes, pull request updates, issue comments, and deployment statuses. This creates a critical visibility gap for development teams who need real-time awareness of repository activity.

This n8n automation solves that problem by connecting GitHub directly to Discord through webhooks. It monitors your repositories for any activity and instantly forwards notifications to your chosen Discord channel. Whether it's a new commit to the main branch, a pull request that needs review, a bug report, or a successful deployment, your entire team gets notified immediately where they're already collaborating.

The workflow intelligently formats notifications with rich Discord embeds, includes relevant links back to GitHub, and can even tag specific team members based on the type of activity. This transforms Discord from just a chat platform into a centralized command center for your development operations.

How It Works

The automation uses GitHub's comprehensive API to monitor repository activity, then formats and delivers those notifications to Discord's webhook system.

Step 1: Monitor GitHub Activity

The workflow periodically checks GitHub's notifications API for new activity across your configured repositories. Unlike GitHub's event webhooks (which don't work consistently for organization repositories), this approach uses the dedicated notifications endpoint that captures all relevant activity.

Step 2: Filter and Process Notifications

Each notification is analyzed and filtered based on your preferences. You can configure the workflow to only forward specific event types (like pushes, pull requests, or issues), ignore certain branches, or focus on particular repositories. The automation extracts key details like the commit message, author, affected files, and links.

Step 3: Format for Discord

The GitHub data is transformed into Discord's rich embed format, creating visually appealing notifications with colored borders (green for success, yellow for pending, red for failures), thumbnails, and organized fields. The formatting makes notifications scannable and actionable.

Step 4: Deliver to Discord Channel

The formatted notification is sent to your Discord webhook URL, appearing instantly in your designated channel. You can configure the workflow to tag specific roles or users—for example, automatically @mentioning your QA team when new bugs are reported or your DevOps team when deployments complete.

Pro tip: To find your Discord user ID for tagging, type \@username#9999 in any Discord channel (replace with your actual username and discriminator). Click your username next to your profile picture to copy the ID, then replace <@userIdForTagging> in the workflow with your actual ID.

What You'll Need

  1. A GitHub account with access to the repositories you want to monitor. Personal access token with appropriate permissions (repo, notifications).
  2. A Discord server where you have permission to create webhooks. Server admin or appropriate channel permissions.
  3. Discord webhook URL created in your target channel settings (Integration → Webhooks → New Webhook).
  4. n8n instance (cloud or self-hosted) to run the automation. The free tier works perfectly for this workflow.
  5. Basic configuration of which repositories and event types to monitor based on your team's needs.

Quick Setup Guide

Follow these steps to implement this automation in under 10 minutes:

  1. Download the template using the button above and import it into your n8n instance.
  2. Create a Discord webhook by going to your Discord channel settings → Integrations → Webhooks → New Webhook. Copy the webhook URL.
  3. Generate a GitHub personal access token with "repo" and "notifications" scopes from GitHub Settings → Developer settings.
  4. Configure the n8n workflow by pasting your Discord webhook URL and GitHub token into the appropriate nodes.
  5. Customize notification filters to specify which repositories, branches, and event types should trigger notifications.
  6. Test the workflow by making a small change to a monitored repository and verifying the Discord notification appears correctly.
  7. Activate the workflow and set it to run automatically (every 5-10 minutes is recommended for real-time notifications).

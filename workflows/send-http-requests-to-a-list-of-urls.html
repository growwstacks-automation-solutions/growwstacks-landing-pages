What This Workflow Does

Manually checking if websites, APIs, or services are running is a time-consuming task that often gets overlooked. This automation solves that problem by systematically sending HTTP requests to a predefined list of URLs at regular intervals—every 15 minutes by default, but configurable to any schedule.

The workflow acts as your digital watchdog, continuously verifying that your critical endpoints are responsive. It's not just about uptime monitoring; this template can be adapted to trigger webhooks, keep free-tier cloud services from going idle, synchronize data between systems, or batch-process requests across multiple APIs without writing a single line of code.

For businesses, this means reduced downtime risk, automated compliance checks, and significant time savings for IT and development teams who would otherwise manually verify services or write custom monitoring scripts.

How It Works

The automation follows a clean, logical flow that's easy to understand and customize for your specific needs.

Step 1: Scheduled Trigger

A schedule node initiates the workflow at your chosen interval—default is every 15 minutes, but you can adjust this to hourly, daily, or any custom cron schedule. This ensures the automation runs consistently without manual intervention.

Step 2: URL List Processing

The workflow reads from a configured list of URLs. Each URL can include protocol, path, and query parameters. The list is maintained within the workflow, making it easy to add or remove endpoints as your needs change.

Step 3: HTTP Request Execution

For each URL in the list, the workflow sends an HTTP request using the configured method (GET by default). It captures the response status, headers, body, and timing metrics, providing comprehensive data about each endpoint's health and performance.

Step 4: Response Handling & Error Management

The workflow processes responses, allowing you to add logic based on status codes. You can extend it to send alerts for failures, log results to a database, or trigger secondary actions based on response content.

Pro tip: Add a "Split Out" node after the HTTP request to separate successful responses (2xx status codes) from failures (4xx/5xx). Route failures to a notification system like Slack or email for immediate alerting.

Quick Setup Guide

Getting started with this automation takes just a few minutes:

  1. Download & Import: Click the download button above to get the JSON file, then import it into your n8n instance via the workflow import function.
  2. Configure Schedule: Adjust the Schedule Trigger node to your desired frequency—keep 15 minutes for active monitoring or set to hourly for less critical endpoints.
  3. Add Your URLs: Replace the sample URLs in the "Set" node with your actual endpoints. Format them with quotes and commas as shown in the example.
  4. Test Run: Execute the workflow once manually to verify all URLs respond correctly and check for any authentication requirements.
  5. Activate & Monitor: Turn the workflow ON and check the execution history to ensure it's running smoothly on your schedule.

Pro tip: Start with a small list of critical URLs and a frequent schedule (5-15 minutes). Once verified, expand to more endpoints and adjust the frequency based on importance—critical services more frequently, informational endpoints less often.

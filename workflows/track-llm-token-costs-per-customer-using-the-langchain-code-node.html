Track LLM Token Costs Per Customer

Automatically monitor and attribute AI usage costs to individual clients for accurate billing and profitability insights.

Download Template JSON · n8n compatible · Free

What This Workflow Does

If you're providing AI-powered services to multiple customers, you're likely burning through LLM credits without knowing which clients are driving your costs. This creates a major business problem: you can't bill accurately, you can't price your services correctly, and you have no visibility into which accounts are actually profitable.

This n8n workflow solves that by automatically tracking every LLM token used and attributing it to specific customers. It captures usage metadata—input tokens, output tokens, model used, timestamps, and customer identifiers—then logs everything to Google Sheets for easy reporting and billing reconciliation.

Think of it as a meter for your AI services. Just like utilities track electricity usage per household, this workflow tracks AI token consumption per customer, giving you the data you need to charge fairly and understand your true cost-to-serve.

Pro tip: This template requires self-hosted n8n because it uses the Langchain Code node, which isn't available in n8n Cloud. Perfect for businesses already running their own n8n instance.

How It Works

The workflow implements a complete token tracking system that captures usage at the point of AI interaction and routes it to your reporting database.

Step 1: Customer Interaction & Data Capture

A form trigger accepts customer submissions (like resume PDFs for conversion). The workflow extracts the file content and captures customer metadata including client ID, project code, and any custom billing parameters you need.

Step 2: AI Processing with Token Tracking

When the Information Extractor node processes the content using an LLM, it uses a custom Langchain Code node as the LLM subnode. This custom node intercepts the token usage metadata through lifecycle hooks, capturing exact token counts before the AI response is even generated.

Step 3: Usage Logging to Google Sheets

The captured token data—along with customer identifiers, timestamps, and cost calculations—gets written to a Google Sheet. This creates a searchable, filterable log of all AI usage across your customer base.

Step 4: Aggregation & Reporting

With all usage data centralized, you can create pivot tables, monthly summaries, and customer-specific reports. The workflow demonstrates how to aggregate this data to understand total costs per client over any time period.

What You'll Need

  1. Self-hosted n8n instance (required for Langchain Code node access)
  2. OpenAI API access or another LLM provider with token-based pricing
  3. Google Sheets for logging usage data (or another database of your choice)
  4. Customer identification system (API keys, user IDs, or account codes to attribute usage)
  5. Basic understanding of n8n workflows for customization and deployment

Pro tip: You can replace Google Sheets with any database (PostgreSQL, MySQL, Airtable) or even send the data to your CRM or billing system via HTTP request. The pattern remains the same—capture metadata, attribute to customer, store for reporting.

Quick Setup Guide

  1. Download and import the template JSON file into your self-hosted n8n instance
  2. Configure your OpenAI credentials in the LLM subnode settings
  3. Set up Google Sheets connection and update the spreadsheet ID in the Google Sheets node
  4. Customize customer metadata capture in the Edit Fields node to match your identification system
  5. Test with sample data to verify token tracking is working correctly
  6. Deploy the workflow and connect it to your customer-facing applications or APIs

For billing purposes, token tracking should be accurate to within 1-2% of actual usage. Most businesses implement rounding to the nearest thousand tokens for simplicity. The key is consistency—using the same measurement method across all customers and billing periods.

Even approximate tracking (90-95% accurate) provides dramatically better insights than no tracking at all. Customers generally accept reasonable approximations when they understand the methodology and see the value correlation between their usage and their bill.

Common mistakes include not accounting for different token prices across models (GPT-3.5 vs GPT-4), forgetting to track both input and output tokens, failing to capture metadata like customer ID or project code, and not setting up alerts for abnormal usage spikes.

Another critical error is storing usage data in a format that's difficult to query for monthly reporting. The workflow addresses this by using structured logging to Google Sheets, but the principle applies to any storage system—design for reporting from the start.

  • Always capture both input and output token counts
  • Include timestamp and model version in every log entry
  • Test with edge cases like failed API calls or retries
